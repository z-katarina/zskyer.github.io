推荐算法小结

Data Mining
  • Algorithm
  • recommendation

推荐系统概述，及常用推荐算法小结。主要参考IBM开发者社区的博文“探索推荐引擎内部的秘密”。

数据源

  • 物品信息：要推荐物品或内容的元数据，例如关键字，基因描述等。
  • 用户信息：系统用户的基本信息，例如性别，年龄等。
  • 用户对物品的偏好：
    • 显式用户反馈：例如用户对物品的评分，对物品的评论。显示的用户反馈能准确的反应用户对物品的真实喜好，但需要用户付出额外的代价。
    • 隐式用户反馈：例如用户购买了某物品，用户查看了某物品的信息等等。隐式的用户行为，数据不精确，而且可能存在较大的噪音，但是只要选择正确的行为特征，通过一些分析和处理，也能得到很好的效果。行为特征的选择可能在不同的应用中有很大不同。例如电子商务网站与音乐网站的特征选取就不一样。

推荐引擎分类

  • 大众行为的推荐引擎：对每个用户都给出同样的推荐，这些推荐可以是静态的由系统管理员人工设定的，或者基于系统所有用户的反馈统计计算出的当下比较流行的物品。
  • 个性化推荐引擎：对不同的用户，根据他们的口味和喜好给出更加精确的推荐，这时，系统需要了解需推荐内容和用户的特质，或者基于社会化网络，通过找到与当前用户相同喜好的用户，实现推荐。
  • 基于人口统计学的推荐：根据系统用户的基本信息发现用户的相关程度。
  • 基于内容的推荐：根据推荐物品或内容的元数据，发现物品或者内容的相关性。
  • 基于协同过滤的推荐：根据用户对物品或者信息的偏好，发现物品或者内容本身的相关性，或者是发现用户的相关性。

其实在现在的推荐系统中，很少有只使用了一个推荐策略的推荐引擎，一般都是在不同的场景下使用不同的推荐策略从而达到最好的推荐效果，例如Amazon的推荐，它将基于用户本身历史购买数据的推荐，和基于用户当前浏览的物品的推荐，以及基于大众喜好的当下比较流行的物品在不同的区域推荐给用户，让用户可以从全方位的推荐中找到自己真正感兴趣的物品。

推荐算法

基于人口统计学的推荐

  • 根据系统用户的基本信息发现用户的相关程度，然后将相似用户喜爱的其他物品推荐给当前用户。
  • 优点：
    1. 因为不使用当前用户对物品的喜欢历史数据，所以对于新用户来讲没有“冷启动”问题。
    2. 该方法不依赖物品本身的数据，所以在不同物品的领域都可以使用。
  • 缺点：
    1. 分类过于粗糙，在对品味要求较高的领域，如图书、电影、音乐等，无法得到很好的推荐效果。
    2. 会涉及到一些与信息发现问题本身无关却比较敏感的信息，有些用户信息不好获取。

基于内容推荐

  • 根据推荐物品或内容的元数据，发现物品或者内容的相关性，然后基于用户以往的喜好记录，推荐给用户相似的物品。
  • 需要对物品进行分析和建模，推荐的质量依赖于对物品模型的完整和全面程度。一般以关键词或标签来描述物品元数据。
  • 物品相似度的分析仅仅依赖于物品本身的特征，没有考虑人对物品的态度。
  • 存在新用户“冷启动”的问题。

基于用户的协同过滤推荐

  • 根据所有用户对物品或者信息的偏好，发现与当前用户口味和偏好相似的“邻居”用户群，一般采用“K-近邻”算法，然后基于这K个邻居的历史偏好信息，为当前用户进行推荐。“喜欢类似物品的用户可能有相似的偏好”
  • 在新闻等领域，物品的个数大于用户的个数，所以一般使用基于用户的协同过滤。

基于物品的协同过滤推荐

  • 根据所有用户对物品或者信息的偏好，发现物品与物品之间的相似度，然后根据用户的历史偏好信息，将类似的物品推荐给用户。
  • 在电商等领域，物品的个数远远小于用户的数量，而且物品的个数和相似度对比比较稳定，所以一般使用基于物品的协同过滤。

协同过滤算法总结

  • 优点：
    1. 不需要对物品或者用户进行严格的建模，而且不要求物品的描述是机器可以理解的，所以该方法也是领域无关的。
    2. 这种方法计算出来的推荐是开放的，可以共用他人的经验，很好的支持用户发现潜在的兴趣偏好。
  • 缺点：
    1. 方法的核心是基于历史数据，所以对新物品和新用户都有“冷启动”问题。
    2. 推荐的效果依赖于用户历史偏好数据的多少和准确性。
    3. 在大部分实现中，用户历史偏好是用稀疏矩阵进行存储的。可能少部分人的错误偏好会对推荐的准确度有很大影响等等。
    4. 对于一些特殊品味的用户不能给予很好的推荐。
    5. 由于以历史数据为基础，抓取和建模用户的偏好后，很难修改或者根据用户的使用演变，从而导致方法不够灵活。

混合推荐机制

  • 加权的混合：用线性公式将几种不同的推荐方法按照一定权重组合起来。
  • 切换的混合：不同场景下，使用不同的推荐方法。
  • 分区的混合：多种推荐机制的结果分为不同的区显示给用户。
  • 分层的混合：将一个推荐机制的结果作为另一个的输入。

协同过滤

收集用户偏好

  • 显示：评分、评论、点赞、转发、收藏
  • 隐式：页面停留时间、购买、查看详情
  1. 分组：将不同的行为分组。
  2. 加权：根据不同行为反映用户喜好的程度，进行加权。一般显示反馈权重大于隐式反馈。
  3. 减噪：通过经典的数据挖掘算法过滤掉行为数据中的噪音，使我们的分析更加精确。
  4. 归一化：不同行为的数据取值可能相差很大，比如用户的查看数据必然比购买数据大很多。需要将各个行为的数据统一在一个相同的取值范围中，从而使得加权求和得到的总体喜好更加精确。最简单的归一化方法是将各类数据除以此类中的最大值，以保证归一化的数据取值在[0，1]中。
  5. 最终得到一个用户偏好二维矩阵，一维是用户列表，另一维是物品列表，值是用户对物品的偏好，一般是[0，1]或者[-1，1]的浮点数值。

相似度计算

  • 基于用户的CF：将一个用户对所有物品的偏好作为一个向量来计算用户之间的相似度。
  • 基于物品的CF：将所有用户对某个物品的偏好作为一个向量来计算物品之间的相似度。

欧几里德距离（距离越小，相似度越大）

皮尔逊相关系数：一般用于计算两个定距变量间联系的紧密程度，它的取值在 [-1，+1] 之间。sx, sy是x和y的样品标准偏差。

相似邻居的计算：

  1. 不论距离远近，固定取最近的K个
  2. 以当前点为中心，取距离为K的圆形区域内的所有点。

计算推荐

  • 基于用户：找到K邻居后，根据邻居的相似度权重以及他们对物品的偏好，预测当前用户没有偏好的未涉及物品，计算得到一个排序的物品列表作为推荐。

  • 基于物品：得到物品的相似物品后，根据用户的历史偏好预测当前用户还没有表示偏好的物品，计算得到一个排序的物品列表作为推荐。

  • 在非社交网络网站中，一般用基于物品的协同过滤推荐。因为用户数量远大于物品的数量，且物品的数据相对稳定，因此计算物品的相似度计算量较小，同时不必频繁更新。在社交网络网站中，一般用基于用户的协同过滤推荐。

推荐多样性

  • 对于单个用户而言，User CF多样性优于Item CF。因为Item CF的推荐就是和以前看的东西最相似的。
  • 从系统的多样性考虑，Item CF多样性优于User CF。因为User CF总是倾向于推荐热门的。
  • 一般情况下应结合两者方法综合推荐。

用户对推荐算法的适应度

  • 对于User CF，推荐的原则是假设用户会喜欢那些和他有相同喜好的用户喜好的东西。因此一个用户对CF算法的适应度和他有多少共同喜好用户成正比。
  • 对于Item CF，推荐原则是假设用户会喜欢和他以前喜欢的东西相似的东西，可以计算一个用户喜欢的物品的自相似度。一个用户喜欢物品的自相似度大，说明他喜欢的东西都是比较相似的，那么他对Item CF的适应度较好。

Slope One

SlopeOne的思想很简单，就是用均值化的思想来掩盖个体的打分差异，举个例子说明一下：
R王五 = 4 - {[(5 - 10) + (4 - 5)] / 2} = 7

用户 对事物A打分 对事物B打分
张三 5 10
李四 4 5
王五 4 ?

当多于两项商品时，计算公式为：
rb = (n (ra - R(a->b)) + m (rc - R(c->b))) / (m + n)
a, b, c代表“商品”，ra代表“商品的打分值”，R(a->b)代表“a组到b组的平均差”，m, n代表人数。
